# CLAUDE.md — Ink

## Project Overview

Ink is a Web3 risk intelligence tool that analyzes smart contracts on Avalanche C-Chain and returns a risk score plus a human-readable explanation generated by AI.

The core idea: blockchain data is public but not understandable. We translate on-chain complexity into simple human decisions.

We are NOT building a block explorer, a trading tool, or a DeFi analytics platform.
We ARE building a layer that makes blockchain risk legible to non-technical users.

## Tech Stack

### Frontend
- Next.js (App Router) + TailwindCSS v4
- Inter font (same as Claude)
- Single-page: hero → ticker → analyzer → how it works
- Language toggle: EN / ES

### Backend
- Node.js + Express
- Single main endpoint: `POST /analyze`
- MongoDB (via Mongoose) for result caching — TTL 1 hour, keyed by address + language
- Port: 8000

### Blockchain — Avalanche C-Chain
- **Snowtrace / Routescan API** — contract verification, source code, ABI
  - URL: `https://api.routescan.io/v2/network/mainnet/evm/43114/etherscan`
- **Routescan API** — token holders (top 1 / top 10 concentration)
  - Endpoints: `tokenholderlist` + `tokensupply`
- **Avalanche RPC** — `owner()` call via ethers.js to detect `ownership_renounced`
  - URL: `https://api.avax.network/ext/bc/C/rpc`

### AI
- **Groq API** — `llama-3.3-70b-versatile`
- Used ONLY for explanation — never for scoring
- Responds in EN or ES based on the `language` parameter

### Storage
- MongoDB Atlas — caches analysis results (address + language → result)
- IPFS: not implemented

## Risk Scoring Engine

### Critical Rule
The LLM does NOT calculate the score. The score is computed by deterministic rules in `backend/src/services/scoringEngine.js`. The LLM only receives the final signals + score and explains them in plain language.

### Signals and Weights

| Signal                     | Condition                    | Points |
|----------------------------|------------------------------|--------|
| Holder concentration top 1 | Top 1 holder > 30%           | +20    |
| Holder concentration top 10| Top 10 holders > 60%         | +15    |
| Ownership                  | Owner not renounced          | +20    |
| Contract verification      | Unverified on Snowtrace      | +15    |
| Mint function              | Present in source code       | +10    |
| Blacklist function         | Present in source code       | +10    |
| Transfer restrictions      | Detected in source code      | +10    |
| Liquidity pool age         | Created < 7 days ago         | +10    |
| Liquidity amount           | < $10,000 USD                | +10    |
| Pause function             | Present in source code       | +5     |
| Is proxy                   | Proxy/delegatecall detected  | +5     |

Score is capped at 100.

### Risk Levels
- 0–30: Low Risk
- 31–60: Medium Risk
- 61–100: High Risk

## AI Prompt Template

```
SYSTEM:
You are a blockchain security analyst. Your job is to explain smart contract risk
to non-technical users in plain, simple language. Be direct. Avoid jargon.
Never invent signals that were not provided. {Respond in Spanish. | Respond in English.}

USER:
Analyze the following smart contract risk data and explain it clearly.

Contract address: {address}
Risk score: {score}/100
Risk level: {level}

Detected signals:
{signals_as_bullet_list}

Your response must include:
1. A 2–3 sentence plain-language explanation of why this contract is risky (or not).
2. The most important warnings, phrased so a non-technical user understands the real-world implication.
3. A one-sentence bottom line.

Do not repeat the raw signal names. Translate them into consequences the user cares about.
```

## Architecture Summary

```
User inputs address
       ↓
POST /analyze (Node.js/Express, port 8000)
       ↓
  ┌────────────────────────────────┐
  │ 1. Validate EVM address        │
  │ 2. Check MongoDB cache         │
  │    (keyed by address+language) │
  └────────────────────────────────┘
       ↓ (cache miss)
  ┌────────────────────────────────┐
  │ Parallel data fetching:        │
  │  • Snowtrace/Routescan API     │
  │    → source code, ABI          │
  │  • Routescan API               │
  │    → holder concentration      │
  │  • Avalanche RPC (ethers.js)   │
  │    → owner() → ownership_renounced │
  └────────────────────────────────┘
       ↓
  ┌────────────────────────────────┐
  │ scoringEngine.js               │
  │  extractSignals() → signals    │
  │  computeScore() → score+level  │
  │  (deterministic, no LLM)       │
  └────────────────────────────────┘
       ↓
  ┌────────────────────────────────┐
  │ Groq API (llama-3.3-70b)       │
  │  → plain-language explanation  │
  │  → EN or ES based on language  │
  └────────────────────────────────┘
       ↓
  Save to MongoDB cache
       ↓
  Return JSON response to frontend
```
